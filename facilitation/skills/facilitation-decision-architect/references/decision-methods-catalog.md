# Decision methods catalog

Detailed working notes on each decision method — when to use, how to run the moment, common failures.

## Leader decides (no input)

### When
- Speed matters more than buy-in
- Leader has full information
- Low-stakes or reversible
- Crisis response

### How to run
1. Leader announces decision
2. Leader explains rationale
3. Leader names owner / next steps

### Common failures
- Used when buy-in is needed → execution problems downstream
- Rationale not shared → mistrust accumulates

### When not to use
Anything that requires the team's commitment. Anything where the team has information the leader doesn't.

---

## Leader decides after consultation

### When
- Leader is accountable
- Team has input that improves the decision
- Buy-in helps but isn't required
- Most leadership decisions of moderate stakes

### How to run
1. Frame: "I'll decide after hearing from each of you. I want your honest input, not a vote."
2. Round-robin input: each person 2–3 minutes
3. Optional: synthesis discussion (15 min)
4. Leader announces decision
5. Leader explains how input shaped (or didn't shape) the call
6. Leader names owner / checkpoint

### Common failures
- Leader pretends it's consensus and then overrides → mistrust
- Leader doesn't explain the call → team can't predict future calls
- Used when team should decide → leader becomes the bottleneck

### Time
30–60 min for consequential decisions

---

## Majority vote

### When
- Simple choice (2–3 options)
- One-person-one-vote is fair (no skill/stake asymmetry)
- Speed matters
- Low-medium stakes

### How to run
1. Frame the options clearly (write them on the wall)
2. Ask for any final clarifying questions
3. "Show of hands for option A... B... C"
4. Announce result
5. Capture in decision record

### Common failures
- Strong minority voice that's right gets overridden
- Trade-offs that need nuance get flattened
- Used as a substitute for genuine deliberation

### Time
15 min from frame to capture

---

## Multi-vote / dot-vote

### When
- Many candidates need narrowing (e.g., 30 ideas → top 5)
- Convergence after divergence
- Want to surface where energy/agreement clusters

### How to run
1. List all candidates visibly
2. Each person gets N votes (rule of thumb: total / 5)
3. People place dots on candidates (sticky dots in person, emoji or vote in Miro)
4. Sort by votes; identify the natural cutoff
5. Discuss the top 3–5 individually before final selection

### Common failures
- Treating it as the final decision (it's a narrowing tool)
- N too high → no convergence
- Allowing strategic voting (people vote against ideas they disagree with strongly rather than for ones they support)

### Variants
- **Weighted dot voting** — different weights for different votes (e.g., 5 dots: one weight 3, two weight 2, two weight 1)
- **$100 voting** — distribute $100 across candidates; forces prioritization
- **Cumulative voting** — multiple votes can stack on one candidate

### Time
20–30 min for ~30 candidates

---

## Fist of five

### When
- Quick consent check on a proposal
- Want to surface disagreement without a long discussion
- Agile teams; recurring decisions

### How to run
1. State the proposal: "I propose we [X]. Fist of five."
2. People show 0–5 fingers:
   - **5** — strong support; love it
   - **4** — support
   - **3** — can live with it; will support
   - **2** — concerns; want to discuss
   - **1** — strong concerns; want to discuss
   - **0** (fist) — block; no, do not proceed
3. If anyone shows 2 or below, discuss their concern before proceeding
4. If anyone shows 0, the proposal is blocked; revise and re-test
5. If everyone shows 3 or above, decision made

### Common failures
- People show 3s without genuine support → quiet dissent later
- Skipping the discussion when 2s appear → resentment

### Time
5–10 min

---

## Consent

### When
- Need workable agreement, not enthusiasm
- "Is there a principled objection?" not "do you all agree?"
- Group is committed to moving forward
- Sociocratic / Holacratic teams

### How to run
1. Proposer states the proposal clearly
2. Clarifying questions only (no objections yet)
3. Reactions round (each person briefly; no debate)
4. Proposer can amend
5. Objection round: "Is there a principled objection — i.e., one that says this is harmful or unworkable?"
6. If no objection, decision passes
7. If objection, integrate (modify proposal to address)

### What counts as a principled objection
- The proposal will cause harm
- The proposal is unworkable
- The proposal is outside agreed scope
- "I prefer something different" is **not** a principled objection — that's preference, not objection

### Common failures
- Treating preference as objection → can't decide anything
- Proposer not pre-validating → many cycles of objection-integrate
- Skipping the reactions round → objections come as surprises

### Time
20–45 min depending on complexity

---

## Consensus

### When
- Decision must have everyone's active agreement (not just consent)
- Small group (≤8)
- High trust
- Time available
- Stakes high enough to justify the cost

### How to run
1. State the proposal
2. Open discussion: each person shares position and reasoning
3. Identify points of agreement and disagreement
4. Iterate the proposal to address disagreement
5. Test for active support: "Can each of you actively support this?"
6. If anyone can't, return to discussion or escalate

### Common failures
- Used at scale → endless deliberation
- Used in low-trust environments → false consensus that breaks later
- Used when consent would do → unnecessary cost

### Time
60–180+ min

---

## RAPID (Bain)

### When
- Cross-functional decisions with multiple stakeholders
- Confusion about who decides vs. who advises
- Recurring decision types

### Roles (assign before the meeting)
- **R**ecommend — the person/group who proposes the decision
- **A**gree — the people whose formal agreement is required (legal, compliance, etc.) — usually 0–2 people
- **P**erform — the people who will execute
- **I**nput — the people who provide expertise but don't decide
- **D**ecide — the single person who makes the call

### How to run
1. Pre-meeting: name the roles for this decision
2. R presents recommendation
3. I roles share input (round-robin)
4. P roles share execution implications
5. A roles confirm or block on their narrow scope
6. D makes the decision
7. Capture and communicate

### Common failures
- Adding roles for politeness ("everyone gets I") → diluted input
- D unclear or shared → no decision lands
- A misused (giving veto power to advisors) → bottleneck

### Time
60–90 min

### Reference
Bain & Company; *Decide & Deliver* (Rogers & Blenko, 2010)

## Integrative decision-making

### When
- Two or more positions seem irreconcilable
- Both positions have legitimate merit
- Hidden integration may be possible
- Team has time to find it

### How to run
1. Each position fully articulated (not just "what" but "why")
2. Identify the underlying need each position serves
3. Ask: "Is there an option neither of us has named that serves both needs?"
4. Generate integrative options
5. Test against both underlying needs
6. Decide via consent or RAPID once integration is found

### Common failures
- Defaulting to compromise (give-each-side-half) instead of integration
- Pushing for integration when none exists → use RAPID and decide

### Time
60–120 min

### Reference
Roger Martin, *The Opposable Mind*; Mary Parker Follett's classic work on integrative bargaining

---

## Trade-off matrix / weighted scoring

### When
- Multiple criteria, comparable options
- Need defensibility (vendor selection, technology choice, hiring)
- Quantitative scoring helps surface trade-offs

### How to run
1. List options as rows
2. List criteria as columns
3. Weight each criterion (1–5 or %, sum to 100)
4. Score each option against each criterion
5. Calculate weighted sum
6. Discuss the result — does it feel right? If not, adjust weights and retest

### Common failures
- Treating the matrix as the answer rather than as input to the conversation
- Picking criteria that conveniently favor a pre-selected option
- Skipping the "does it feel right?" sanity check

### Time
60–90 min for a substantive decision

## Choosing the right method — quick decision tree

```
Is the decision irreversible or high-stakes?
├─ Yes → invest in deliberation
│        Is execution buy-in critical?
│        ├─ Yes → Consent or RAPID
│        └─ No → Leader decides after consultation
└─ No → use a faster method
         Is one person holding the picture?
         ├─ Yes → Leader decides
         └─ No → Multi-vote (narrow) → Fist of five (confirm)
```
