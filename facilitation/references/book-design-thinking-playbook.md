# Book digest: *The Design Thinking Playbook*

**Authors**: Michael Lewrick, Patrick Link, Larry Leifer (Stanford Center for Design Research, Hasso Plattner DT Research Program). Foreword by Larry Leifer.
**What this book is**: A reference manual for facilitating design thinking — both the canonical Stanford-d.school cycle (problem → empathy → focus → ideate → prototype → test) and the *organizational* extensions (transforming teams, designing ecosystems, integrating with strategic foresight, AI, and data analytics).

The book uses Tetris-block iconography to visually mark its sections; the structure is modular and you can drop into any chapter standalone.

## The Design Thinking cycle (canonical Stanford form)

Although the book uses ten how-to chapters in Part 1, they map to the canonical Stanford d.school cycle:

```
1. Understand the problem (1.3 problem statement)
2. Empathize with users (1.4 needs, 1.5 empathy)
3. Define the focus (1.6 focus)
4. Ideate (1.7 generate, 1.8 structure/select)
5. Prototype (1.9)
6. Test (1.10)
```

The cycle is iterative; failure on test loops you back to define or ideate.

## Key facilitator moves the book teaches

These are the moves to extract for the `facilitation-design-thinking-coach` skill:

- **Problem reframing** — almost every misfire in DT is a poorly-stated problem. The facilitator's first job is to challenge the framing.
- **Empathy interviews and observation** — structured listening with users, paired with observation. The facilitator coaches the team through the discomfort of going wide before narrowing.
- **Persona and Empathy Map work** — externalize the user view so the team can argue with the artifact, not with each other.
- **Point of View statements** — the bridge from research to ideation; "User X needs Y because Z."
- **HMW (How Might We) prompts** — ideation kickers framed as questions.
- **Idea selection structure** — most teams default to consensus or HiPPO; the book gives explicit selection structures.
- **Prototype fidelity matching** — match prototype investment to learning goal. Paper for early concepts, digital for usability, working for technical feasibility.
- **Test design** — what you're testing for (desirability, feasibility, viability) determines who you test with and how.

## Visual signature

The book's **Tetris-block iconography** marks chapter types and sequencing. It's a teaching device — design thinking as block-stacking, where the order matters but you can recombine. Worth mentioning when teaching DT to a Slalom team that hasn't seen the book.

## How to reach for this in real work

- **Designing a discovery sprint inside a Slalom Summit Discover phase** — Part 1, especially 1.4–1.6 (needs, empathy, focus).
- **Coaching a client team into adopting DT** — Part 2, especially 2.5 (triggering change as facilitator) and 2.6 (preparing the organization).
- **Strategy + DT engagements (the Composable DXP sweet spot)** — Part 3, especially 3.1 (systems thinking), 3.2 (lean business model), 3.3 (ecosystem design).

## How this book relates to others in this plugin

- **Design a Better Business** is the canvas-driven cousin; pair them when running strategy + DT engagements.
- **Collaboration Code Tools** has many DT-compatible modules (Empathy Map, Empathic Interview, Hassle Map, Prototyping, etc.) — the Playbook gives the *why* and the cycle; Tools gives the *how* of each module.
- **Meeting Design (Hoffman)** covers the meeting cadence around DT work — kickoff meetings, daily check-ins, retros.
